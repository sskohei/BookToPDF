# アーキテクチャ

## 概要

BookToPDF は**クライアント完結の Web アプリ**である。サーバーサイドのアプリケーションコードは持たず、
ビルド成果物は静的ファイルとしてホスティングされる。画像処理・PDF生成はすべてユーザーのブラウザ内で行われ、
撮影した写真が端末外（サーバー等）に送信されることはない。

```
┌──────────────────────────── ブラウザ ────────────────────────────┐
│                                                                    │
│  Capture UI ──▶ 画像取り込み ──▶ Web Worker ──▶ OpenCV.js (WASM)  │
│  (カメラ/ファイル)                  │                              │
│                                      ▼                              │
│                              補正済み画像 (Mat → Blob/DataURL)     │
│                                      │                              │
│                                      ▼                              │
│                    CornerEditor（手動微調整・確認UI）              │
│                                      │                              │
│                                      ▼                              │
│                    PageReorder（並び替え・削除・回転）             │
│                                      │                              │
│                                      ▼                              │
│                         pdf-lib で PDF 組み立て                    │
│                                      │                              │
│                                      ▼                              │
│                            ダウンロード (Blob URL)                 │
└────────────────────────────────────────────────────────────────────┘
```

フロントエンドはNext.js（App Router）で構築するが、`output: 'export'` による静的エクスポートを使い、
API Routes/SSRなどのサーバー機能は利用しない。カメラ/ファイル入力・OpenCV.js・Web Workerを扱うコンポーネントは
すべてクライアントコンポーネント（`"use client"`）とし、サーバーコンポーネントは静的なレイアウト・マークアップ
部分のみに限定する。サーバーは静的ファイル配信のみを担当する（Vercel / Cloudflare Pages / GitHub Pages 等）。

## なぜクライアント完結なのか

- **プライバシー**: 本のページ写真という私的な画像データを外部サーバーに送信しない。
- **運用コスト**: サーバー・ストレージ・画像処理用の計算資源を持たないため、運用コストがほぼゼロ。
- **オフライン利用**: PWA化することで、通信環境に依存せず撮影〜PDF化まで完結できる。

トレードオフとして、処理速度は端末の性能に依存する。低スペック端末での速度劣化は許容し、
将来的に処理が重すぎる場合のみサーバーサイド処理の追加を再検討する（現時点では非採用）。

## 画像処理パイプライン

1枚の写真に対して、以下の順で処理を行う（実装は `frontend/src/lib/cv/` に置き、Web Worker から呼び出す）。

1. **読み込み**: 画像を OpenCV Mat に変換。処理は縮小版で行い、最終出力には元解像度を使用する。
2. **四隅検出**: グレースケール化 → GaussianBlur → Canny エッジ検出 → `findContours` で最大の四角形輪郭を抽出し、
   本のページ境界候補とする。`approxPolyDP` が返す4頂点をそのまま使わず、輪郭全体（間引かれる前）の
   4辺それぞれに直線をフィットし、隣接する辺同士の交点として頂点位置を精密化する（`contourGeometry.ts`の
   `refineQuadCorners`）。これにより輪郭近似のノイズ・角の丸まりに起因する台形補正のズレを抑える。
   木目調の机など質感のある背景では classical CV(Canny/Otsu)ベースの検出が誤りやすいことが実写真での
   検証で繰り返し確認されており、将来的にはこのステップを ONNX Runtime Web 経由の軽量なDLモデルに
   置き換える案を検討中（未採用、`docs/tech-stack.md` 参照）。採用する場合も、モデルのロード失敗・
   低信頼度時は本ステップの classical CVパイプラインにフォールバックする設計を維持する。
3. **フォールバック**: 四隅が検出できない場合は自動処理を諦め、手動調整UI（CornerEditor）にゆだねる。
   自動検出は失敗しうる前提で必ず手動修正の経路を用意する。
4. **見開き判定**: 検出領域のアスペクト比が単ページ想定より横長であれば見開き候補とし、
   中央付近の輝度プロファイルの谷、または Hough変換による縦線検出で綴じ目（gutter）位置を特定して確定する。
5. **見開き分割**: gutter位置で画像を左右に分割し、それぞれ独立に四隅を再検出する。
6. **透視変換・湾曲補正**: 単ページ・見開き分割後のそれぞれの領域に対し、まず綴じ目付近の上下辺の
   湾曲（本を開いたときの物理的な曲面によって生じる、平面のホモグラフィでは直せない歪み）が有意かを
   判定する。有意な場合は上下辺それぞれに二次曲線をフィットし、両曲線間をルールドサーフェスで
   つないだ`cv.remap`による変形（`dewarpPage`）で補正する。湾曲データが無い、または有意でない
   （フラットなページ、または手動調整で4頂点のみ指定された場合）は、従来通り`getPerspectiveTransform`
   + `warpPerspective`による矩形（台形補正済み）画像への変換にフォールバックする。
7. **傾き補正 (deskew)**: 変換後に残る傾きを Hough変換等で検出し、回転補正する。個々の線を絶対角度で
   足切りするのではなく、線同士の合意（暫定中央値との一致）に基づいてロバストに角度を推定するため、
   四隅検出の精度不足などで実際に生じた大きめの残存傾きも補正できる。
8. **画質補正**: CLAHE によるコントラスト強調、明るさ補正、余白トリミングを行う。
9. **確認・手動調整**: 処理結果をプレビューし、ユーザーが四隅ハンドルのドラッグで微調整・回転・削除できる
   オーバーレイUIを提供する。

## PDF生成

- 確定したページ画像を、ユーザーが指定したページ順に `pdf-lib` で PDF の各ページとして埋め込む。
- 画質とファイルサイズのバランスは JPEG 品質パラメータで調整する。
- 生成した PDF は Blob URL 経由でブラウザからダウンロードさせる。

## データフロー上の注意点

- 画像データ（Mat, Blob, DataURL）はすべてブラウザのメモリ/IndexedDB内に留まる。外部への送信経路は存在しない。
- 重い処理（OpenCV.js の呼び出し全般）は必ず Web Worker 上で実行し、メインスレッド（UI）をブロックしない。
- ページ配列・処理ステータスなどのアプリ状態は `frontend/src/state/` で管理し、UIコンポーネントと画像処理ロジックの間の
  唯一の受け渡し経路とする。

## リポジトリレイアウト

Next.js プロジェクト一式は `frontend/` サブディレクトリに置く（設計ドキュメントは `docs/` にリポジトリ直下で並置）。
これは将来的にフロントエンド以外の要素（ドキュメントサイトや補助スクリプト等）をリポジトリに追加する余地を
残すための構成であり、クライアント完結という方針そのものには影響しない。開発コマンドはすべて `frontend/` 配下で
実行する（詳細は `docs/setup.md`）。
