# 開発への参加ガイド

複数人での開発を見据えた最小限のルール。プロジェクトの状況に応じて更新していく。

## ブランチ運用

- `main` を常にデプロイ可能な状態に保つ。
- 作業は機能単位のブランチを `main` から切って行い、Pull Request 経由で `main` にマージする。
- ブランチ名は内容が分かる形にする（例: `feature/gutter-detection`, `fix/corner-detection-fallback`）。

## コミット規約

- 1コミットは意味のある単位にまとめる（無関係な変更を1コミットに混ぜない）。
- コミットメッセージは変更の「何を」だけでなく「なぜ」が分かるようにする。
- 画像処理ロジック（`src/lib/cv/`）を変更した場合は、対応するテストの追加・更新を同じPRに含める。

## Pull Request

- PRの説明には、変更内容・動作確認方法（実機確認を行った場合はその端末/ブラウザ）を記載する。
- 画像処理・UIフローに関わる変更は、可能な限り `docs/setup.md` の手順でローカル起動し、
  実際に写真をアップロードして動作確認したことを明記する。
- `docs/roadmap.md` のフェーズをまたぐような大きな変更を行う場合は、事前にIssue等で方針を共有する。

## コーディング規約

`CLAUDE.md` の「開発時の規約」を参照。要点:

- TypeScriptを使用し `any` を避ける。
- UIコンポーネントと画像処理ロジック（`lib/cv`, `lib/pdf`）を分離する。
- 重い画像処理は Web Worker 経由で実行する。
- 画像処理ロジックの変更にはVitestのユニットテスト、UIフローの変更にはPlaywrightのE2Eテストを添える。

## ドキュメントの更新

- 技術選定を変更した場合は `docs/tech-stack.md` を更新する。
- アーキテクチャやパイプラインの流れを変更した場合は `docs/architecture.md` を更新する。
- 実装が進みフェーズが完了したら `docs/roadmap.md` の状態を更新する。
- ドキュメントと実装が乖離しないよう、実装を伴うPRでは関連ドキュメントの更新もあわせて行う。
