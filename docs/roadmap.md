# ロードマップ

BookToPDF の実装は以下の3フェーズで進める。各フェーズは前フェーズが動く状態になってから着手する。

## Phase 1: MVP

目標: 画像アップロードからPDF出力までの一連の流れを、簡易な精度で通す。

- 画像アップロード（ファイル選択）
- 自動四隅検出（`findContours` ベース）
- 透視補正（台形補正）
- 見開き分割は**簡易版**: 画像中央で固定的に2分割し、それぞれに透視補正を適用する
  （gutterの自動検出は Phase 2 に持ち越す）
- 検出結果の手動微調整UI（四隅ドラッグ）
- 複数ページの並び替え・削除
- pdf-lib によるPDF書き出し・ダウンロード

## Phase 2: 精度向上

目標: スキャンアプリ相当の補正品質に引き上げる。

- gutter（綴じ目）自動検出による本格的な見開き判定・分割
- 傾き自動補正（deskew）
- コントラスト補正（CLAHE）・明るさ補正・余白トリミング
- PWA化（オフライン動作・ホーム画面インストール）
- カメラ入力（`getUserMedia`）によるアプリ内カメラ撮影フロー

## Phase 3: 発展

目標: 実運用での使い勝手を高める。

- 連続撮影フロー（1ページ撮ったら自動で次のページ撮影に移る等）
- ページ並び替えUIの高度化（サムネイル一覧での一括編集など）
- OCR（将来検討。採用する場合はクライアント完結の方針を維持できるか改めて検討する）
- DLベースの四隅検出の検討（将来検討。ONNX Runtime Webで軽量モデル(DocAlignerベースを軸に検討)を
  Web Worker内で動かし、既存のclassical CVパイプラインへのフォールバックを維持する構成を想定。
  詳細は `docs/tech-stack.md` の「四隅検出のDL化」を参照。フルページdewarpingへのDL導入は
  モデルサイズ・ライセンス制約から対象外とする）

## フェーズを跨ぐ方針

- 各フェーズの完了条件は「実機（iOS Safari / Android Chrome）で一連の流れが動作すること」とする。
- 精度向上（Phase 2）は実装が複雑になりやすいため、Phase 1 で固定分割による見開き処理を先に動かし、
  段階的にgutter検出へ置き換える。
- サーバーサイド処理の追加は、クライアント完結では処理精度・速度が明らかに不十分と判明した場合のみ検討する
  （`docs/tech-stack.md` の「採用しなかった選択肢」を参照）。
